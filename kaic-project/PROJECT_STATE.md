# KAIC Project State

Документ для продолжения разработки в новом чате без потери контекста.
Актуален на момент консервации — все факты ниже соответствуют реальному
состоянию кода в архиве, а не намерениям или планам.

---

## 1. Общая идея проекта

**Цель KAIC (Kirpich AI Core):** личная локальная агентная система для
одного пользователя (не SaaS, не фреймворк, не продукт для многих людей).
Должна помогать: разрабатывать открытый проект Kirpich MIND, вести
инженерные проекты, монтировать видео, отслеживать новости — и в
перспективе стать "локальной командой ИИ-агентов", а не просто чат-ботом.

**Главный принцип разработки**, соблюдавшийся на всём протяжении проекта:
*"Если задачу можно решить 50 строками вместо 500 — выбираем 50."*
Никакого Big Design Up Front, сложность добавляется только по факту
реальной, а не гипотетической боли.

**Текущая архитектура (как есть, реализовано):**

```
Electron UI (React+TS+Vite)
        ↓  HTTP (poll, 1 сек)
Control Center API (axum, Rust)
        ↓
Task Store (SQLite)  ←→  Router (детерминированный, без LLM)
        ↓                        ↓
   Scheduler  ←──────── Capability Registry (YAML)
        ↓                        ↑
  ModelBackend (trait)  ←── Resource Registry (YAML)
        ↓
  LmStudioBackend (рабочий) | EmbeddedBackend (todo!()-каркас)
        ↓
     LM Studio → llama.cpp
```

**Что уже реализовано:** полный вертикальный срез — от создания задачи в
Electron UI до её выполнения через LM Studio и обратно, включая
Continue/Pause/Cancel. Реализован **на уровне кода**, но **ни разу не
прогнан целиком вживую** (см. раздел "Известные ограничения" — это важно).

**Что ещё не реализовано:** Agent/Tool framework (вся система сейчас
устроена как "текст → категория → модель → текст"), Telegram Bridge,
любые специализированные агенты (Video/Browser/CAD/EmbedIDE), реальный
рабочий `EmbeddedBackend` (сейчас — пустой каркас), поле `model` в `Task`,
содержимое `StatusBar`, миграции схемы SQL.

## 2. Архитектура — по компонентам

### Router (`router.rs`)
Полностью детерминированный, **не вызывает LLM и не знает моделей**.
`Category` — закрытый `enum` (Simple, Programming, LargeProject,
Engineering, DeepAnalysis, Video, Research). `Router::classify(text,
metadata) -> Category` — порядок проверки: (1) `TaskMetadata.agent_category`
если задан агентом → используется без анализа; (2) метаданные размера
(`file_count`/`total_lines`) → `LargeProject`; (3) ключевые слова по
категориям (от специфичных к общим); (4) default → `Simple`. Возвращает
только `Category::as_str()` — строку, дальше в системе категория везде
трактуется как `&str`/`String`, не как enum.

### Capability Registry (`capability_registry.rs`)
Простая YAML-таблица: `HashMap<String, CapabilityEntry{primary, fallback:
Vec<String>, manual_only: bool}>`. Никакой логики выбора — просто данные.
`candidates(category)` возвращает упорядоченный список моделей-кандидатов.
`MaximumQuality`/`Qwen40B` — единственная категория с `manual_only: true`.

### Resource Registry (`resource_registry.rs`)
Тоже простая YAML-таблица, но про **стоимость**, не про способности:
`ResourceEntry{vram_mb, ram_offload_mb, load_seconds, always_loaded,
preferred_device: PreferredDevice(Gpu|Cpu|Hybrid), path: Option<String>}`.
`path` добавлен позже — для будущего `EmbeddedBackend` (путь к `.gguf`),
у LM Studio-моделей не используется. `always_loaded_vram_mb()` и
`always_loaded_models()` — для Scheduler'а.

### Scheduler (`scheduler.rs`)
Диспетчер ресурсов GPU, не знает о содержании задач. `run(category,
allow_manual, request) -> GenerateResponse` — единственная публичная точка
входа: держит `gpu_lock: Mutex<()>` на **весь** цикл (подбор модели +
загрузка/вытеснение + сама генерация) — сознательное решение сериализовать
всю систему в обмен на простоту (альтернатива — счётчик `in_use` на
модель — рассмотрена и отложена). LRU-вытеснение никогда не трогает
`always_loaded` модели. `preload_always_loaded()` проверяет, что
always_loaded-модели вообще влезают в `total_vram_mb`, прежде чем
пытаться их грузить.

### ModelBackend (`model_backend.rs`)
`trait ModelBackend: Send+Sync { is_loaded, load, unload, generate }` —
не упоминает HTTP/LM Studio нигде в сигнатурах. `Message{role: Role, content}`,
`Role{System,User,Assistant}`, `GenerateRequest{messages}`,
`GenerateResponse{content}`.

**`LmStudioBackend`** — рабочая реализация. `load`/`unload`/`is_loaded`
через нативный LM Studio REST API v1 (`/api/v1/models*`), `generate` через
OpenAI-совместимый `/v1/chat/completions`. Хранит `instance_ids:
Mutex<HashMap<String,String>>` — LM Studio не гарантирует, что
`instance_id == имя модели`, поэтому запоминается реальный `instance_id`
из ответа `load()` и используется при `unload()` (фикс из аудита).

**`EmbeddedBackend`** (`embedded_backend.rs`) — **пустой `todo!()`-каркас**,
компилируется только за Cargo-фичей `embedded-backend`/`embedded-backend-cuda`.
Реальной логики нет специально — см. раздел про EmbeddedBackend ниже.

### Task Store (`task_store.rs`)
SQLite через `rusqlite`, доступ — `Arc<std::sync::Mutex<Connection>>`,
все операции через `tokio::task::spawn_blocking`. `TaskStatus` — Running /
WaitingForHuman / Paused / Done / Failed / Cancelled (с `Serialize`/
`Deserialize`, snake_case). `ContextEntry{role: String, content: String,
at: DateTime<Utc>}` — хранится как YAML-блоб в одной `TEXT`-колонке, SQL
не знает внутреннюю форму. `Task{id, category, status, created_at,
updated_at, context, pending_telegram_message_id}`. Методы: `create`,
`get`, `list`, `set_status`, `set_pending_telegram_message`,
`append_context`. Схема создаётся через `CREATE TABLE IF NOT EXISTS` —
**механизма миграции нет** (известный технический долг, см. ниже).

### Control Center API (`control_center_api.rs`)
Единый файл — транспорт (axum) + маршруты вместе, разделять было не на
что. `AppState{task_store, scheduler, resource_registry}`. Маршруты:
`GET/POST /tasks`, `GET /tasks/:id`, `POST /tasks/:id/continue|pause|cancel`,
`GET /models`, `GET /status`. `create_task`/`continue_task` не блокируют
HTTP-запрос — сохраняют/обновляют задачу немедленно и запускают
`run_task_pipeline` в `tokio::spawn` фоном; `continue_task` пересобирает
**полную** историю через `context_to_messages()`, а не только последнее
сообщение. `TaskRouter` — алиас для `crate::router::Router` (конфликтовал
с `axum::Router`, поймано в аудите). CORS — `permissive()` (локальный
инструмент, один пользователь).

### Electron UI (`kaic-ui/`)
React + TypeScript + Vite, Electron — только окно, никакого IPC/preload
(renderer общается с backend напрямую через `fetch()`). Без Redux/Zustand/
Context — общее состояние только там, где реально понадобилось (два
случая, оба в `App.tsx`): `pendingTasks` (оптимистичные плейсхолдеры) и
`selectedTask`+`selectedTaskId` (общие для `Workspace`+`DetailsPanel`,
чтобы не дублировать `GET /tasks/{id}`). Поллинг — простой `setInterval`,
без WebSocket, без `usePolling`-абстракции (не было второго места, которое
оправдало бы её выделение). Компоненты: `Header` (форма создания
задачи), `TaskList` (список, клик = выбор), `Workspace` (история задачи +
форма Continue), `DetailsPanel` (метаданные + Pause/Cancel), `StatusBar`
(**так и остался пустой заглушкой**).

## 4. Design Review — итоговые выводы (все 5 блоков)

Проведён после того, как первый вертикальный срез был реализован в коде,
**но до его реального ручного прогона** (важно — см. ограничения).

### Блок 1. Category
**Вопрос:** остаётся ли `Category` хорошим фундаментом, или `enum` станет
ограничением?
**Вывод: Да.**
**Аргументация:** `enum` используется только внутри `Router::classify()`.
Везде ниже (API, Capability Registry, Scheduler, Task) категория уже
`String`. Явный путь для будущих агентов (агент сам объявляет свою
категорию через API) уже работает без изменений кода — нужно только
добавить запись в YAML. Трение есть только для **автоклассификации**
новых категорий по свободному тексту — там нужна правка `router.rs`
(~15 строк) и перекомпиляция, плюс `TaskMetadata.agent_category` типизирован
как `Option<Category>`, а не `Option<String>` (минимальный путь убрать —
поменять этот один тип).

### Блок 2. Scheduler
**Вопрос:** является ли Scheduler сервисом, готовым стать инструментом
агента, или он жёстко привязан к "вызову модели"?
**Вывод: Да.**
**Аргументация:** `Scheduler::run()` уже узко специализирован —
`(category, allow_manual, messages) → response`, не знает о `Task`, не
управляет жизненным циклом. "Директор", который сегодня безусловно
считает, что любая задача заканчивается вызовом LLM — это
`run_task_pipeline()` в `control_center_api.rs`, не сам `Scheduler`.
Стоимость перехода: заменить/обернуть `run_task_pipeline` диспетчером
по категориям (`match category { "Video" => ..., _ => default_llm_pipeline }`),
ноль изменений в `scheduler.rs`. `gpu_lock` — per-call, не per-task, так
что агент может звать `Scheduler` несколько раз вперемешку с другими
инструментами без ручного управления блокировкой.

### Блок 3. Context
**Вопрос:** можно ли превратить `Context` из истории переписки в
универсальный журнал событий без переделки TaskStore?
**Вывод: Да, самый дешёвый переход из всех.**
**Аргументация:** `ContextEntry` хранится как YAML-блоб в одной
`TEXT`-колонке — SQL не знает внутреннюю форму. Добавление типизированных
полей (`tool_call`, `artifact` и т.д.) через `#[serde(default)]` не требует
изменений ни в SQL-схеме, ни в `TaskStore`, ни в API — только правка одной
Rust-структуры + инкрементальная доработка рендеринга на фронтенде.
**Честная оговорка:** эта дешевизна — плата за то, что мы изначально
выбрали blob вместо нормализованной таблицы; нельзя SQL-запросом найти
"все задачи с tool_call к ffmpeg" без разбора каждого блоба.

### Блок 4. Task → Mission
**Вопрос:** можно ли позже добавить иерархию (`parent_task_id`), не
переписывая фундамент?
**Вывод: Да** — с находкой, не специфичной для Mission.
**Аргументация:** `parent_task_id: Option<Uuid>` — аддитивное поле, Router
и Scheduler о `Task` вообще не знают, изменения нужны только в
структуре/API/UI (по тому же паттерну, что уже использовался для
`pending_telegram_message_id`).
**Реальная находка:** `TaskStore` не имеет **никакого механизма миграции
схемы** (`CREATE TABLE IF NOT EXISTS` не трогает уже существующую
таблицу). Это всплывёт при **любом** будущем изменении схемы, не только
из-за Mission — зафиксировано как отдельный технический долг.

### Блок 5. ModelBackend
**5a. Архитектурная совместимость — Да, полностью.**
Trait не упоминает транспорт нигде; единственное место, знающее про LM
Studio за пределами `model_backend.rs` — одна строка в `main.rs`. Замена
= новый `impl ModelBackend` + одна строка.

**5b. Стоимость миграции — умеренная, сконцентрирована в одном месте.**
Токенизация, sampling, загрузка GGUF — бесплатно (ядро самого биндинга).
Streaming не нужен нам сейчас (UI поллинговый) — не часть стоимости.
Единственный значимый риск — **chat-темплейт**: `llama.cpp` умеет
полноценный Jinja2 (`minja`) и это подтверждено конкретным вызовом API
(`model.apply_chat_template(None, messages, true)` в `llama-cpp-4`), но
качество зависит от того, насколько версия `llama.cpp` внутри биндинга
свежая для конкретной новой архитектуры модели (найден реальный
баг-репорт — устаревший `llama-cpp-2` 0.1.146 падал на chat-темплейте
Gemma-4).

**5c. Стоимость поддержки и сборки:**
Поддержка — разумный компромисс (обновления через `cargo update`, изредка
подгонка кода под смену мажорной версии биндинга) — не "поддерживать свой
рантайм", но и не буквально бесплатно.
Сборка — **самый жёсткий и однозначный вывод всего Review**: у всех
проверенных активно поддерживаемых Rust-биндингов (`llama-cpp-2/4`,
`llama_cpp`/edgenai, `knap-ai`, `intentee`) CUDA-фича требует установленный
CUDA Toolkit **на машине сборки**, не только запуска. Сейчас `cargo build`
KAIC не требует ничего, кроме Rust toolchain — это изменится. CPU-сборка
остаётся доступна как фолбэк, но лишает смысла offload на RTX 4060. Найден
частичный смягчающий фактор — `LLAMA_PREBUILT_DIR` в форке
`eugenehp/llama-cpp-rs`, позволяющий пропустить компиляцию при наличии
готовых бинарников (никто их пока не публикует под нашу конфигурацию).

**Итог Design Review:** все пять блоков — **Да**. Фундамент подтверждён
как допускающий эволюцию к Agent/Tool Framework и к `EmbeddedBackend` без
переписывания `Router`/`Scheduler`/`TaskStore`/API.

## 5. EmbeddedBackend — что решили и почему

**Что решили:** не удалять LM Studio и не трогать `LmStudioBackend`.
Вместо этого создать **параллельную** реализацию `ModelBackend` —
`EmbeddedBackend` — поверх `llama-cpp-4` (обёртка над `llama.cpp`, не
собственный inference engine).

**Почему `llama-cpp-4`:** после Design Review и отдельного технического
исследования (см. ответы на чек-лист в истории чата) — активно
поддерживается (v0.4.2, `llama.cpp` b9982 — июль 2026, включает свежие
апстрим-фичи), используется в продакшене (utilityai), `apply_chat_template`
подтверждён конкретным примером API, поддерживает CUDA/batched
inference/несколько одновременно загруженных моделей/unload через `Drop`.

**Почему отказались сразу интегрировать:** документация ≠ проверенный
код. Часть сигнатур API (в частности — параметризован ли `LlamaModel`
временем жизни `LlamaBackend`, точные имена методов `LlamaBatch`/
`LlamaSampler`) собрана по фрагментам поиска, а не через реальную
компиляцию (в среде разработки Claude нет Rust toolchain и самой
библиотеки). Риск: обернуть непроверенный API сразу в `trait`+
`spawn_blocking`+`HashMap`-архитектуру и потом одновременно чинить и
баги библиотеки, и архитектуру.

**Почему сначала `experiments/llama_cpp_smoke`:** полностью автономный
Cargo-проект (свой `Cargo.toml`, ноль зависимости от `kaic-backend`),
чтобы:
1. проверить своими руками, а не по документации: сборку, CUDA, GGUF,
   chat-темплейт, генерацию;
2. не тянуть транзитивные зависимости KAIC (`axum`/`rusqlite`/`teloxide`)
   в проверку CUDA-сборки;
3. не поддерживать два источника правды — пока эксперимент не доказал
   себя, `embedded_backend.rs` — это осознанно пустой `todo!()`-каркас,
   а не черновик с частичной логикой, которая начала бы расходиться
   с экспериментом при каждой правке.

**Cargo-фичи** (`embedded-backend`, `embedded-backend-cuda`) введены,
чтобы обычный `cargo build` для `kaic-backend` не начал молча требовать
CUDA Toolkit — обычная разработка (Router/Scheduler/UI) не должна
деградировать из-за экспериментальной ветки.

**Статус эксперимента:** код `experiments/llama_cpp_smoke/src/main.rs`
написан, но **ни разу не скомпилирован и не запущен** — это первая
задача для нового чата.

## 6. Roadmap (порядок следующих этапов)

1. **Прогнать `experiments/llama_cpp_smoke`** — сначала CPU-сборка
   (`cargo run -- /путь/model.gguf`), затем CUDA (`cargo run --features
   cuda -- ...`). Проверить глазами промпт после `apply_chat_template` и
   скорость генерации (ток/сек).
2. Починить то, что не соберётся с первого раза — три помеченных в коде
   риска (см. раздел 5) наиболее вероятные кандидаты.
3. Перенести рабочий код из эксперимента в `embedded_backend.rs`, заменив
   `todo!()`. На этом же шаге явно решить стратегию `LlamaContext`
   (пересоздавать на каждый `generate()`, как в первом срезе, или
   переиспользовать между вызовами ради инкрементального KV-cache) — этот
   вопрос сознательно отложен до появления рабочего кода.
4. Параллельно/до этого — **завершить так и не проведённый ручной прогон**
   исходного вертикального среза на LM Studio (Electron + kaic-backend +
   реальный LM Studio) — см. "Известные ограничения", это не потеряло
   актуальность просто потому, что фокус сместился на EmbeddedBackend.
5. Подключить `EmbeddedBackend` в `main.rs` как альтернативный
   `Arc<dyn ModelBackend>` (механизм выбора между backend'ами — CLI-флаг,
   конфиг или ветка по Cargo-фиче — не спроектирован, нужно решить).
6. Прогнать `EmbeddedBackend` через реальный Scheduler/TaskStore/API/UI
   целиком.

## 7. Все принятые архитектурные решения (без пропусков)

- Простота важнее универсальности; сложность добавляется только по факту
  реальной боли, не заранее.
- Single-GPU сериализация — осознанный выбор для одного пользователя на
  одной видеокарте, не недостаток.
- Router — полностью детерминированный (regex/keywords/метаданные), без
  вызова LLM/embedding — на RTX 4060 8GB цена подгрузки классификатора
  слишком высока.
- Capability Registry и Resource Registry — простые YAML-таблицы без
  scoring engine, весов, ML — модели уже протестированы вручную, что они
  умеют — известно заранее.
- L0-L5 — ось стоимости, не интеллекта; многомерный Capability Profile
  обсуждался, но **не реализован** — вместо него простой список
  primary+fallback по категории, осознанно проще.
- Router определяет только категорию, никогда не имя модели; Capability
  Registry сопоставляет категорию с моделью; Scheduler решает вопрос
  ресурсов — строгое разделение по всему стеку.
- `MaximumQuality`/Qwen40B — только вручную, никогда не выбирается
  автоматически.
- Task Store обязан быть персистентным (SQLite) — переживает перезапуски,
  это основа Human-in-the-Loop.
- Telegram отложен на вторую итерацию; первый HITL-клиент — Electron через
  локальный HTTP API (axum), не через IPC/napi-rs/stdio-RPC.
- Поллинг (1 сек) вместо WebSocket для первого прототипа — пересмотреть
  только если реально станет проблемой.
- Транспорт и маршруты Control Center API — один файл, разделять было не
  на что.
- Без Redux/Zustand/Context во фронтенде — только `useState`+пропсы;
  состояние поднимается в `App.tsx` только когда реально нужно шарить
  (два случая за весь проект: `pendingTasks`, `selectedTask`).
- Оптимистичное обновление UI при создании задачи — сопоставление по
  реальному `id` от backend'а (`realTaskId`), не по тексту и не по
  таймеру — оба альтернативных подхода были реализованы, найдены
  проблемы, заменены.
- `React.memo` и другая перф-оптимизация — не добавляется до реальной
  замеренной/увиденной проблемы.
- Design Review — фиксированный процесс из 5 блоков с критерием
  Да/Нет+аргументация+стоимость+путь миграции, проводится **до** любых
  новых архитектурных решений; установлен как стандартный цикл: "этап →
  аудит → следующий этап".
- `EmbeddedBackend` — только параллельная реализация trait'а, ничего
  существующего (`Scheduler`/`Router`/`TaskStore`/API/Electron/
  `LmStudioBackend`) не меняется на этом этапе.
- Доказывать технологию полностью автономным экспериментом до интеграции
  в KAIC — не поддерживать два расходящихся источника правды одновременно.
- Никакого самописного inference engine/GGUF-парсера/KV-cache с нуля —
  явно отвергнуто; используется только зрелая существующая библиотека
  через Rust-биндинги.

## 8. Известные ограничения

- **Самое важное: ни разу не проведён полный ручной прогон системы**
  (Electron + kaic-backend + реальный LM Studio) целиком, вживую. Весь
  код проверен через аудит/чтение, но не через реальное выполнение.
  Пользователь начал готовиться к этому прогону, затем переключил
  приоритет на исследование `EmbeddedBackend` **до** того, как прогон был
  завершён. Это означает: возможны баги, не пойманные статическим
  анализом (реальные имена моделей в LM Studio, поведение сети, реальные
  тайминги).
- `experiments/llama_cpp_smoke` написан, но не скомпилирован и не
  запущен — три места в коде явно помечены как непроверенные.
- `Scheduler.run()` сериализует всю систему одной блокировкой на весь
  цикл (подбор + генерация) — пока одна задача генерирует ответ, все
  остальные ждут, даже если могли бы использовать другую уже загруженную
  модель. Осознанный компромисс, не баг.
- Независимый поллинг `TaskList` и `selectedTask`(`App.tsx`) — возможна
  рассинхронизация статуса до ~1 секунды между списком и панелью деталей.
- Ошибка `GET /tasks/{id}` при обновлении открытой задачи проглатывается
  молча — остаются последние известные данные, индикатора ошибки нет.
- `Task` не хранит, какую модель реально выбрал Scheduler — `DetailsPanel`
  показывает только категорию.
- `TaskStore` не имеет механизма миграции схемы SQL.
- `StatusBar` — не реализован, пустая заглушка с самого начала проекта.
- Agent/Tool Framework не существует вообще — вся система сейчас
  однородна: "текст → категория → модель → текст". Design Review
  подтвердил, что это можно надстроить, но ничего не реализовано.
- Имена моделей в `capability_registry.yaml`/`resource_registry.yaml` —
  мнемоники (`Fable9B`, `GLM47` и т.д.), требуют ручной сверки с реальными
  именами моделей в LM Studio пользователя — не автоматизировано,
  и не подтверждено, что сверка была сделана.
- CUDA-сборка `EmbeddedBackend` потребует установленный CUDA Toolkit на
  машине сборки — не только запуска.
